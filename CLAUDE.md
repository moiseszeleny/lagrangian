# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

`feynlag` ‚Äî a pure-SymPy library that derives tree-level Feynman rules from user-written BSM (Beyond Standard Model) Lagrangians. Input style is FeynRules-like: declare particle fields with gauge/discrete representations, write Lagrangian terms explicitly with library building blocks (`Dmu`, `dag`, `Bilinear`), and the library checks invariance, breaks the symmetry, diagonalizes, and extracts vertices.

## Architecture

### Pipeline stage ‚Üí module map

The library is organized around one linear physics pipeline; each stage lives in its own module and the top-level `Model` (in `lagrangian.py`) orchestrates all of them **lazily** (`functools`-style caching via `Model._cache`, invalidated on `solve_tadpoles`/`rotate` ‚Äî nothing is computed at construction or at import time, deliberately, to avoid a known flaw in the DLRSM1 reference implementation this library was built from):

1. **Declare** ‚Äî `parameters.py` (`ExternalParameter` fixed by experiment, `InternalParameter` derived later, `ParameterSet` topologically sorts internal dependencies), `fields.py` (`Scalar`/`Fermion`/`GaugeBoson`, auto-generated component symbols + conjugate registry), `groups/` (`gauge.py`: U1/SU2/SU3 with explicit generator matrices; `discrete.py`: ZN/S3 with Clebsch‚ÄìGordan products).
2. **Write** ‚Äî `Lagrangian`/`LagrangianTerm` (`lagrangian.py`) collect sector-tagged terms (`kinetic`, `potential`, `yukawa`, `gauge`, `other`); `operators.py` provides `Dmu` (covariant derivative) and `PartialMu` (the `‚àÇ_Œº` placeholder, resolved to momentum space later).
3. **Check** ‚Äî `invariance.py`: gauge invariance by infinitesimal component transformation (`Œ¥œÜ = iŒ±^aT^aœÜ`), discrete invariance by finite generator substitution, hermiticity, mass-dimension power counting. `Model.check_invariance()` is the entry point.
4. **Break the symmetry** ‚Äî `vacuum/ewsb.py` (`Vacuum`: VEV shift maps `œÜ‚Å∞‚Üí(v+h+ia)/‚àö2`), `vacuum/tadpoles.py` (extract + solve, registers solutions on `InternalParameter`s), `vacuum/masses.py` (mass matrices: real, complex/charged via Dummy-conjugate trick, gauge-boson).
5. **Diagonalize** ‚Äî `vacuum/diagonalize.py`: `Rotation` (weak‚Üîphysical substitution + verification), 2√ó2 analytic (`tan 2Œ∏`), SVD (Dirac), Takagi (Majorana).
6. **Extract vertices** ‚Äî `vertices/extract.py` (Poly-based commuting-symbol extractor, bosons only), `vertices/bilinear.py` (`Bilinear` ‚Äî the fermion-sandwich track, separate from the extractor since fermions aren't commuting symbols), `vertices/yangmills.py` (cubic/quartic pure-gauge self-couplings from structure constants, not from the extractor), `vertices/vertex.py` (`Vertex` objects classified into the closed catalog SSS/SSSS/VSS/VVS/VVSS/VVV/VVVV/FFS/FFV).
7. **Export** ‚Äî `export/latex.py`, `export/ufo/` (writer + UFO `object_library`/`lorentz_map`/`pycode` for MadGraph-importable model directories; `vvvv.py` assembles the 4-boson self-coupling into the 3 UFO Lorentz structures, and colored (SU(3)) vertices get real `T`/`f` color-tensor strings ‚Äî see writer.py's `add_vvv_vertex`/`add_vvvv_vertex`/`add_fermion_vertex`).

`dirac.py` (Clifford algebra, chiral projectors `diracPL`/`diracPR`) and `verify/checks.py` (dual symbolic+numeric verification utilities, `numeric_equal`) are used across stages.

### The two-track extraction design (important, easy to get wrong)

Bosonic fields (scalars, gauge bosons) are represented as plain commuting `sympy.Symbol`s and go through the `Poly`-based extractor in `vertices/extract.py`. Fermion fields are `IndexedBase`-typed (one flavor-indexed component per generator), and a fermion sandwich `œàÃÑŒìœà` is wrapped in the opaque `Bilinear(bar_indexed, gamma, field_indexed)` atom ‚Äî this **never** enters the commuting-symbol extractor. `vertices/bilinear.py`'s `extract_fermion_vertices` groups terms by `(bar, gamma, field)` and peels boson legs off the coefficient with the bosonic extractor.

**Four-fermion (FFFF) operators** (dim-6 effective operators like the Fermi muon-decay operator `(œàÃÑŒìœà)(œáÃÑŒì‚Ä≤œá)`) are the **two-bilinear** case of the same track (`extract_fermion_vertices` accepts one *or* two `Bilinear` factors; a nested `((bar,Œì,field),(bar',Œì‚Ä≤,field'))` sorted-pair key, distinguished from the flat one-bilinear key by `isinstance(key[0], tuple)`). Two design commitments, both pinned in `test_four_fermion.py`: (1) the **as-written bilinear basis** ‚Äî no Fierz canonicalisation, a Fierz-rearranged operator is a *different input* (matches SMEFT-UFO practice); (2) **four distinct fermion components only** ‚Äî a repeated component (incl. a squared bilinear `B¬≤`, which `atoms()` would dedup, so `_bilinear_factors` counts multiplicity by walking factors) raises `NotImplementedError`, because identical legs generate cross-chain Wick contractions whose relative signs need genuine spinor-index Fierz algebra the opaque-`Bilinear` design cannot express. With distinct legs there are no exchange contractions, so the rule is the plain scalar `i¬∑coeff¬∑‚àè(boson mult)!` (`four_fermion_feynman_rule`) with the two Dirac structures `(Œì,Œì‚Ä≤)` carried **separately** ‚Äî folding them into one product would wrongly invoke the Clifford algebra *across* chains (contract a shared `Œ≥^Œº`). A vector-current contraction `(œàÃÑŒ≥^ŒºP_Lœà)(œáÃÑŒ≥_ŒºP_Lœá)` is written with a **shared index symbol**, `DiracGamma(mu)` on one leg and `DiracGammaLower(mu)` on the other (`dirac_conjugate` handles the lowered `Œ≥_Œº` ‚Äî self-conjugate like `Œ≥^Œº` since the metric is real). The mass-dimension check gains a `max_dim` flag (`Model.check_invariance(max_dim=6)` / `validate(max_dim=6)`) to admit dim-6 operators; with the Wilson coefficient carrying its physical dim ‚àí2 the *term* is still dim-4, so the flag matters only under the operator-dimension convention (dimensionless coefficient). UFO export: 8 `FFFF*` Lorentz structures (S‚äóS / V‚äóV with a contracted Lorentz dummy) in `lorentz_map.py`, `writer.add_four_fermion_vertex` (carries the Feynman-rule `i`), `write_ufo(four_fermion_vertices=‚Ä¶)`. **Export the h.c. vertex too**: a Hermitian `op + h.c.` is two FFFF vertices, and MadGraph needs the conjugate pair to route fermion-number flow through a contact interaction (a UFO with only one ‚Üí `NoDiagramException`; this is a MadGraph requirement, not a feynlag one ‚Äî the exported Lorentz string is verbatim the one MadGraph's shipped `taudecay_UFO` uses). Validated end-to-end: `scripts/madgraph_fermi.py` reproduces the muon width `Œì(Œº‚ÜíeŒΩŒΩ)=G_F¬≤m_Œº‚Åµ/192œÄ¬≥` to 0.07% (`docs/benchmark.md`, not in CI). Worked model: `examples/fermi_theory.py`.

Because `Bilinear` and `PartialMu` are custom SymPy `Function`s with no `fdiff` rule, code that needs to differentiate an expression containing one of them **must never let the differentiation variable end up inside their arguments** ‚Äî SymPy's chain rule falls back to an unevaluated, non-cancelling `Subs(Derivative(...))` placeholder otherwise. This has caused two real bugs (both fixed): gauge-invariance checking of `Dmu`-built kinetic terms, and of fermion `Bilinear` terms. The fix pattern in both cases: exploit that the transform commutes with the operator (‚àÇ_Œº or the flavor index) and substitute *outside*, or explicitly redistribute the operator over `Add` args first (see `_transform_map`/`_fermion_transform`/`_expand_bilinear` in `invariance.py`, and `D_linear` in `operators.py`) ‚Äî never call `sp.diff` on an expression where `PartialMu`/`Bilinear` wraps a variable-dependent argument.

Related gotcha: `Expr.as_coeff_Mul()` defaults to `rational=True` and only pulls out *Rational* coefficients, silently leaving Dummy/Symbol factors bundled into the "atom" side. Any code splitting `coeff * atom` terms (as the bilinear-distribution logic does) must find the atom explicitly via `as_ordered_factors()`, not rely on `as_coeff_Mul()`.

**Fermion mass-basis rotations** work through the ordinary `Rotation`/`Model.rotate` machinery with `Indexed` old-fields (e.g. `Rotation([eL[i], EL[i]], [e1L[i], e2L[i]], rotation_2x2(Œ∏))`) ‚Äî register **two rotations per chirality** (field-side AND bar-side, same matrix), and write every Lagrangian term with the same flavor-index symbol so the `xreplace` keys match. The resulting `Add`-valued legs inside `Bilinear` slots are distributed by `expand_bilinear` (`vertices/bilinear.py`, moved from invariance.py), which `extract_fermion_vertices` and `fermion_mass_matrix` apply automatically ‚Äî without it, extraction silently grouped by unsplit composite keys. For symbolic 2√ó2 Dirac (biunitary/SVD) diagonalization use `diagonalize_svd_2x2` (Œ∏_L from `M¬∑M·µÄ`, Œ∏_R from `M·µÄ¬∑M`, analytic tan-2Œ∏ route) ‚Äî `diagonalize_svd`'s generic `.diagonalize()` path returns unusable nested radicals for symbolic entries. `DiracFermion` is unsupported and **raises on construction** (a 4-component path would need its own physics-validated mass/diagonalization sector) ‚Äî model vector-like fermions as two `WeylFermion`s with identical reps (see `examples/sm_vll.py`). Note `dirac_conjugate` strips a stray `diracI` factor (so a vector current `Œ≥^Œº¬∑diracI` conjugates cleanly).

Fermion-bilinear **hermiticity** is checked via `Bilinear._eval_conjugate` (`vertices/bilinear.py`), which implements `(œàÃÑ‚ÇÅŒìœà‚ÇÇ)‚ÄÝ = œàÃÑ‚ÇÇŒìÃÑœà‚ÇÅ` with `ŒìÃÑ = Œ≥‚Å∞Œì‚ÄÝŒ≥‚Å∞` computed by `dirac.dirac_conjugate` (covers `diracI`, `diracPL`/`diracPR`, bare `DiracGamma(mu)`, and `DiracGamma(mu)*diracPL`/`diracPR` ‚Äî raises `NotImplementedError` for anything else, e.g. products of ‚â•2 gamma matrices, rather than guessing) and a `bar_partner` registry (`fields.py`) that every `Fermion` populates at construction to find "the bar of this field"/"the field of this bar". A Yukawa term written without its `+ h.c.` partner now genuinely fails `Model.check_invariance()`'s hermiticity check; a gauge current (`fermion_gauge_current`) is hermitian by construction with no hand-written h.c. needed.

**Discrete-symmetry invariance for fermion multiplets** (`ZN`/`S3` acting on fermion generations ‚Äî e.g. two lepton doublets forming an S‚ÇÉ doublet, matching `thdm_s3.py`'s scalar S‚ÇÉ pattern but for leptons) is also checked, via `DiscreteSymmetry.fermion_generator_data()` (`groups/discrete.py`) + `_fermion_transform_discrete` (`invariance.py`). Unlike the gauge case, discrete transformations are *finite* (no `Œ±`-linearization/differentiation needed) ‚Äî a multiplet member substitutes directly as `œà'_i = Œ£_k M[i,k]œà_k`. The bar leg uses `X = (M‚Åª¬π)·µÄ`, **not** `M` itself (derived from requiring `Œ£_i œàÃÑ'_iœà'_i` invariant, verified by direct computation, not assumed) ‚Äî `X` only coincides with `M` when `M` is real orthogonal (true for `S3`'s irreps, not `ZN`'s complex-phase ones, where `X = MÃÑ`). `assign()` rejects multiplets mixing `Fermion` and non-`Fermion` members (they'd need different substitution mechanisms ‚Äî flat dict vs. index-preserving `.replace()`).

### Model.validate() umbrella and the charge/hermiticity vertex checks

`Model.validate()` (`lagrangian.py`) aggregates every applicable check into one `ValidationReport`: `check_invariance`, `check_anomalies` (`anomalies.py`, skipped without charged fermions), the UFO round-trip (`verify/ufo_roundtrip.py::verify_ufo_numeric`, only with a `ufo_path`), and ‚Äî when a declared `charges` map + physical `fields` are passed ‚Äî the three `charges.py` checks. **Electric charge is never hardcoded as `Q=T3+Y`**: `derive_charge_operator` solves for the diagonal-generator combination annihilating every VEV (null-space; excludes groups no VEV'd scalar transforms under, so colour/spectator U(1)s drop out; raises on rank-0 "charge-breaking vacuum" and rank-2 "ambiguous" cases), and `physical_charges` propagates weak‚Üíphysical charges through the registered `Rotation`s as `Q_new = R¬∑Q_old¬∑R‚Åª¬π` (this is what makes `W1/W2 ‚Üí W¬±` come out ¬±1 automatically). The declared map is the authority for the vertex checks; the derivation is the cross-check (`check_charge_consistency`, scale fixed by the *most common* declared/derived ratio so a single mistuned declaration is flagged, not the anchor). `check_hermiticity_pairing` needs the partner coupling `(-1)^{1+d}¬∑conj(coupling)` where **d = momentum-tag degree** (0 ‚Üí ‚àíconj, 1 ‚Üí +conj: the VSS acid test ‚Äî derived empirically, pinned in `test_charges.py`), and requires an explicit full `conjugates` antiparticle map because charged physical fields like `W¬±` come from rotations, not `conjugate_pair` (so they're absent from `conjugate_map`).

### Quark flavour / CKM (`flavor.py`)

CKM is done the FeynRules-SM way, **never** a symbolic 3√ó3 Yukawa SVD (generic complex Yukawas don't diagonalize in closed form ‚Äî nested radicals): work in the quark **mass basis** with diagonal Yukawas and **insert** V into the charged current only (`g/‚àö2 W‚Å∫ ≈´_i Œ≥^ŒºP_L V_ij d_j`), leaving neutral currents flavour-diagonal (GIM). `standard_ckm()` builds V from the exact PDG standard parametrization (3 angle externals + 1 phase + 9 complex-internal `V_ij`), unitary as a trig identity so `V‚ÄÝV` simplifies to `1` (a matrix of 9 independent symbols is NOT unitary and would fake Z-FCNCs). CKM elements must be **scalar** parameters (`ParameterSet` is scalar-only; the UFO printer emits an invalid `V[0,1]` subscript for an `IndexedBase`); the 9 `V_ij` export as `type='complex'` internals and round-trip. The 2-gen Cabibbo case IS derivable through the ordinary real `Rotation`/`rotation_2x2` machinery ‚Äî `test_ckm.py::TestTwoGenerationGIM` pins the GIM cancellation (mixing in W, exact `cos¬≤+sin¬≤` diagonality in Z). Worked demo: `examples/sm_ckm.py`.

### MadGraph round-trip (the exported UFO actually computes right)

`scripts/export_sm_ufo.py` + `scripts/madgraph_roundtrip.py` export an EW+leptons SM UFO and compare cross sections to MadGraph's stock `sm` at a matched parameter point (`e+e-‚ÜíŒº+Œº- = 2.788 pb`, `e+e-‚ÜíW+W- = 19.50 pb`, both to MC precision; `docs/benchmark.md`). Not in CI (each launch compiles Fortran). MG5 lives at `~/.local/mg5dl/MG5_aMC_v3_7_2` and runs on Python 3.13. **Three UFO-export issues this surfaced** (first two fixed in the library + regression-tested, invisible to unit tests but fatal in MG): (1) generated UFO files must use **absolute** imports (`from object_library import ‚Ä¶`, not `from .object_library`) ‚Äî MG runs them as standalone scripts for param-card generation; `verify/ufo_roundtrip.py` + the test `_import_ufo` helpers now load a UFO the MG way (dir on `sys.path`, import submodules by bare name). (2) `add_fermion_vertex` must apply the **Feynman-rule `i`** to the L/R couplings (the bosonic/VVV paths already carry it) ‚Äî omitting it is invisible in `e+e-‚ÜíŒºŒº` (overall phase cancels in |M|¬≤) but breaks FFV‚ÜîVVV interference in `e+e-‚ÜíWW` (gives 98 pb not 19.5). (3) The triple-gauge coupling from the **complex W¬± rotation** comes out of `cubic_couplings` with the opposite overall sign to MG's VVV1 convention ‚Äî the *export script* flips `gAWW`/`gZWW` (validated by the WW round-trip); the real-basis `ggg=-g_s` is unaffected and already matches stock, so this is NOT a universal `add_vvv_vertex` negation.

### SU(3) / QCD vertex dynamics

The group theory (`SU3` Gell-Mann generators/structure constants, `groups/gauge.py`) and the vertex-extraction machinery (`Dmu`, `fermion_gauge_current`, `cubic_couplings`/`quartic_couplings`) were already fully generic over gauge group ‚Äî no SU(3)-specific code was needed there. What closed the gap: `examples/sm_scalar_gauge.py` now includes a quark sector (SU(2) doublet `QL` + `SU3c`-triplet singlets `uR`/`dR`, flavor-generic `Yu`/`Yd` Yukawas mirroring the lepton pattern ‚Äî 3 generations, undiagonalized, no CKM: CKM mixing is physically orthogonal to SU(3) vertex dynamics and a fully generic 3-generation complex-Yukawa SVD doesn't resolve in closed symbolic form) and a gluon field; `export/ufo/vvvv.py`'s `assemble_vvvv` builds the 3-structure VVVV1/2/3 decomposition of the 4-boson self-coupling that `yangmills.py`'s `quartic_couplings()` docstring had deferred as unbuilt "Phase 5" (derived by direct functional differentiation of `-¬ºF^aF^a`, cross-checked against both SU(2) and SU(3) in `tests/test_yangmills.py`); `export/ufo/writer.py` emits real color-tensor strings (`T(3,1,2)` for qqg, `f(1,2,3)` for ggg, three `f*f` structures for gggg) instead of the hardcoded singlet `'1'` ‚Äî gluons are exported as **one** UFO particle repeated (color summed via the tensor string), never as 8 separate weak-basis components.

### Conventions (all pinned by tests ‚Äî see `CONVENTIONS.md` for the full list)

Metric `(+,‚àí,‚àí,‚àí)`; `P_L=(1‚àíŒ≥‚ÇÖ)/2`; `D_Œº=‚àÇ_Œº‚àíigT^aA^a_Œº`; VEV expansion with explicit `1/‚àö2`; Feynman rule = `i √ó coefficient √ó ‚àè(field multiplicity)!`; momentum convention `‚àÇ_ŒºœÜ‚Üíip(œÜ)œÜ`; simplification hierarchy `expand‚Üícollect‚Üífactor‚Üísimplify` (simplify last); positive-assumption dummies for anything under `sqrt`; every physical result gets dual verification (symbolic diff **and** `verify.numeric_equal` random-point check).

## Tests and examples

Tests are organized per-module/per-physics-topic (`test_invariance.py`, `test_fermion_sector.py`, `test_scalar_pipeline_sm.py`, `test_gauge_sector_sm.py`, `test_ufo_export.py`, `test_qcd.py`, `test_yangmills.py`, `test_ufo_qcd.py`, `test_vll.py`, etc.) and pin actual physics values (e.g. `hWW = igm_W`, `m_h¬≤=2Œªv¬≤`, `ggg=-g_s`), not just code paths ‚Äî when adding a feature, add a test that pins the expected physical result, following the existing pattern in the relevant `test_*.py` file. `test_qcd.py` pins the SU(3) sector (gauge invariance of a color-triplet current, qqg, ggg, gggg); `test_yangmills.py` is the group-generic ground-truth derivation of the VVVV quartic self-coupling assembly (SU(2) and SU(3)); `test_ufo_qcd.py` checks the emitted UFO color-tensor strings/values; `test_vll.py` pins the vector-like-lepton mixing physics (RH-only Z-FCNC, h-coupling sum rule, SM decoupling); `test_u1x.py` pins the SM√óU(1)_X / Z‚Ä≤ physics (symbolic-charge invariance, 3√ó3 neutral gauge matrix, tan 2Œ∏‚Ä≤, Goldstone counting, Z‚Ä≤ff couplings, B‚àíL limit); `test_four_fermion.py` pins the FFFF track (Fermi-operator extraction/rule, the distinct-legs/Fierz guard, two-bilinear charge+hermiticity checks, the `max_dim` EFT flag, and the `FFFF*` UFO round-trip).

`examples/` contains full worked models used as both documentation and manual smoke tests: `sm_scalar_gauge.py` (complete SM: Higgs + electroweak gauge + lepton sector + quark/QCD sector), `sm_vll.py` (SM + vector-like lepton doublet: bare Dirac mass, biunitary 2√ó2 diagonalization, fermion mass-basis rotations, modified Z/W/h couplings), `sm_u1x.py` (SM √ó U(1)_X with X = aY + b(B‚àíL) kept **symbolic**: X-charged Higgs ‚Üí tree-level Z‚ÄìZ‚Ä≤ mixing via two *chained* 2√ó2 rotations ‚Äî Weinberg first, then Z‚ÄìZ‚Ä≤ consuming the intermediate `Z0` symbol, legal because `physical_lagrangian` applies rotations sequentially in registration order; singlet scalar + Higgs portal; 3 ŒΩR with Dirac Yukawa via HÃÉ; no UFO export ‚Äî UFO's particle table calls `float()` on charges), `thdm.py` (2HDM), `thdm_s3.py` (3HDM with S‚ÇÉ flavor symmetry ‚Äî the tadpole system there is deliberately over-constrained, forcing a vacuum alignment), `fermi_theory.py` (muon-decay four-fermion operator: the FFFF track end-to-end ‚Äî dim-6 operator with `max_dim=6`, extraction, U(1)_em invariance below the EW scale, `FFFF*` UFO export), and `SM_Feynman_Rules_Tutorial.ipynb` / `SM_VLL_Tutorial.ipynb` / `SM_U1X_Tutorial.ipynb` (all *executed* notebooks ‚Äî regenerate their outputs with `jupyter nbconvert --execute` after any edit that could change results, don't hand-edit output cells; the VLL notebook walks `sm_vll.py` stage by stage, including a standalone demo of why `expand_bilinear` is needed for fermion mass-basis rotations).

The notebooks are tracked through a `nbstripout` git filter (`.gitattributes`, `*.ipynb filter=nbstripout diff=ipynb`), configured with `--keep-output` so real outputs (plots, printed Feynman rules) stay in git history and in diffs ‚Äî only `execution_count` and volatile per-cell metadata get normalized away on `git add`/`git diff`, so re-executing a notebook without changing its actual results produces no diff noise. A fresh clone needs to activate the filter once (`.gitattributes` alone only declares the pattern, it doesn't install the filter driver):

```bash
nbstripout --install --attributes .gitattributes
git config filter.nbstripout.clean "$(git config filter.nbstripout.clean) --keep-output"
git config diff.ipynb.textconv "$(git config diff.ipynb.textconv) --keep-output"
```

(`nbstripout --install` itself does not persist CLI flags like `--keep-output` into the stored filter command ‚Äî they have to be appended to `filter.nbstripout.clean`/`diff.ipynb.textconv` by hand, as above.)
