# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

`feynlag` — a pure-SymPy library that derives tree-level Feynman rules from user-written BSM (Beyond Standard Model) Lagrangians. Input style is FeynRules-like: declare particle fields with gauge/discrete representations, write Lagrangian terms explicitly with library building blocks (`Dmu`, `dag`, `Bilinear`), and the library checks invariance, breaks the symmetry, diagonalizes, and extracts vertices.

## Architecture

### Pipeline stage → module map

The library is organized around one linear physics pipeline; each stage lives in its own module and the top-level `Model` (in `lagrangian.py`) orchestrates all of them **lazily** (`functools`-style caching via `Model._cache`, invalidated on `solve_tadpoles`/`rotate` — nothing is computed at construction or at import time, deliberately, to avoid a known flaw in the DLRSM1 reference implementation this library was built from):

1. **Declare** — `parameters.py` (`ExternalParameter` fixed by experiment, `InternalParameter` derived later, `ParameterSet` topologically sorts internal dependencies), `fields.py` (`Scalar`/`Fermion`/`GaugeBoson`, auto-generated component symbols + conjugate registry), `groups/` (`gauge.py`: `U1` + `SUN` for any SU(N), with `SU2`/`SU3` as thin subclasses keeping the familiar names/legacy labels; `discrete.py`: ZN/S3 with Clebsch–Gordan products). SU(N) generators for **any irrep** are built dynamically in the Gelfand–Tsetlin basis (`groups/sun.py`) — no hardcoded matrices beyond σ/λ. Rep labels are an int *dimension* (`1`→trivial, `N`→fundamental, `N²−1`→adjoint, other ints resolved by the Weyl dimension formula, raising on ambiguous dims like 15 of SU(3)), a **conjugate** label (negative int `−d` or string `"dbar"`, giving `T̄^a=−(T^a)^*`), or an explicit **Dynkin tuple** `(a_1,…,a_{N−1})`. Every irrep of a group shares the fundamental's structure constants `f^{abc}` (ρ is a homomorphism), so `[T^a,T^b]=i f^{abc}T^c` and the Dynkin-index normalization `Tr(T^aT^b)=S(R)δ^{ab}` are fixed by the algebra, not tuned — this is what lets `Dmu`/`fermion_gauge_current`/`yangmills` stay generic over the rep. `test_sun.py` pins the SU(4)/SU(5) algebra, the 6 of SU(3) (`C_2=10/3`, shared-`f`), conjugates, and the anomaly-free SU(5) `5̄+10` generation.
2. **Write** — `Lagrangian`/`LagrangianTerm` (`lagrangian.py`) collect sector-tagged terms (`kinetic`, `potential`, `yukawa`, `gauge`, `other`); `operators.py` provides `Dmu` (covariant derivative) and `PartialMu` (the `∂_μ` placeholder, resolved to momentum space later).
3. **Check** — `invariance.py`: gauge invariance by infinitesimal component transformation (`δφ = iα^aT^aφ`), discrete invariance by finite generator substitution, hermiticity, mass-dimension power counting. `Model.check_invariance()` is the entry point.
4. **Break the symmetry** — `vacuum/ewsb.py` (`Vacuum`: VEV shift maps `φ⁰→(v+h+ia)/√2`), `vacuum/tadpoles.py` (extract + solve, registers solutions on `InternalParameter`s), `vacuum/masses.py` (mass matrices: real, complex/charged via Dummy-conjugate trick, gauge-boson).
5. **Diagonalize** — `vacuum/diagonalize.py`: `Rotation` (weak↔physical substitution + verification), 2×2 analytic (`tan 2θ`), SVD (Dirac), Takagi (Majorana).
6. **Extract vertices** — `vertices/extract.py` (Poly-based commuting-symbol extractor, bosons only), `vertices/bilinear.py` (`Bilinear` — the fermion-sandwich track, separate from the extractor since fermions aren't commuting symbols), `vertices/yangmills.py` (cubic/quartic pure-gauge self-couplings from structure constants, not from the extractor), `vertices/vertex.py` (`Vertex` objects classified into the closed catalog SSS/SSSS/VSS/VVS/VVSS/VVV/VVVV/FFS/FFV).
7. **Export** — `export/latex.py`, `export/ufo/` (writer + UFO `object_library`/`lorentz_map`/`pycode` for MadGraph-importable model directories; `vvvv.py` assembles the 4-boson self-coupling into the 3 UFO Lorentz structures, and colored (SU(3)) vertices get real `T`/`f` color-tensor strings — see writer.py's `add_vvv_vertex`/`add_vvvv_vertex`/`add_fermion_vertex`).

`dirac.py` (Clifford algebra, chiral projectors `diracPL`/`diracPR`) and `verify/checks.py` (dual symbolic+numeric verification utilities, `numeric_equal`) are used across stages.

### The two-track extraction design (important, easy to get wrong)

Bosonic fields (scalars, gauge bosons) are represented as plain commuting `sympy.Symbol`s and go through the `Poly`-based extractor in `vertices/extract.py`. Fermion fields are `IndexedBase`-typed (one flavor-indexed component per generator), and a fermion sandwich `ψ̄Γψ` is wrapped in the opaque `Bilinear(bar_indexed, gamma, field_indexed)` atom — this **never** enters the commuting-symbol extractor. `vertices/bilinear.py`'s `extract_fermion_vertices` groups terms by `(bar, gamma, field)` and peels boson legs off the coefficient with the bosonic extractor.

**Four-fermion (FFFF) operators** (dim-6 effective operators like the Fermi muon-decay operator `(ψ̄Γψ)(χ̄Γ′χ)`) are the **two-bilinear** case of the same track (`extract_fermion_vertices` accepts one *or* two `Bilinear` factors; a nested `((bar,Γ,field),(bar',Γ′,field'))` sorted-pair key, distinguished from the flat one-bilinear key by `isinstance(key[0], tuple)`). Two design commitments, both pinned in `test_four_fermion.py`: (1) the **as-written bilinear basis** — no Fierz canonicalisation, a Fierz-rearranged operator is a *different input* (matches SMEFT-UFO practice); (2) **four distinct fermion components only** — a repeated component (incl. a squared bilinear `B²`, which `atoms()` would dedup, so `_bilinear_factors` counts multiplicity by walking factors) raises `NotImplementedError`, because identical legs generate cross-chain Wick contractions whose relative signs need genuine spinor-index Fierz algebra the opaque-`Bilinear` design cannot express. With distinct legs there are no exchange contractions, so the rule is the plain scalar `i·coeff·∏(boson mult)!` (`four_fermion_feynman_rule`) with the two Dirac structures `(Γ,Γ′)` carried **separately** — folding them into one product would wrongly invoke the Clifford algebra *across* chains (contract a shared `γ^μ`). A vector-current contraction `(ψ̄γ^μP_Lψ)(χ̄γ_μP_Lχ)` is written with a **shared index symbol**, `DiracGamma(mu)` on one leg and `DiracGammaLower(mu)` on the other (`dirac_conjugate` handles the lowered `γ_μ` — self-conjugate like `γ^μ` since the metric is real). The mass-dimension check gains a `max_dim` flag (`Model.check_invariance(max_dim=6)` / `validate(max_dim=6)`) to admit dim-6 operators; with the Wilson coefficient carrying its physical dim −2 the *term* is still dim-4, so the flag matters only under the operator-dimension convention (dimensionless coefficient). UFO export: 8 `FFFF*` Lorentz structures (S⊗S / V⊗V with a contracted Lorentz dummy) in `lorentz_map.py`, `writer.add_four_fermion_vertex` (carries the Feynman-rule `i`), `write_ufo(four_fermion_vertices=…)`. **Export the h.c. vertex too**: a Hermitian `op + h.c.` is two FFFF vertices, and MadGraph needs the conjugate pair to route fermion-number flow through a contact interaction (a UFO with only one → `NoDiagramException`; this is a MadGraph requirement, not a feynlag one — the exported Lorentz string is verbatim the one MadGraph's shipped `taudecay_UFO` uses). Validated end-to-end: `scripts/madgraph_fermi.py` reproduces the muon width `Γ(μ→eνν)=G_F²m_μ⁵/192π³` to 0.07% (`docs/benchmark.md`, not in CI). Worked model: `examples/fermi_theory.py`.

**Majorana bilinears / the dim-5 Weinberg operator** (`MajoranaBilinear` in `vertices/bilinear.py`) are the same-chirality, charge-conjugation-contracted sandwich `ψ₁ᵀ C Γ ψ₂` — **both legs are field components** (no bar), the middle `Γ` **must** contain the charge-conjugation matrix `diracC` (`dirac.py`, `C=iγ²γ⁰`, with `Cᵀ=C†=C⁻¹=−C`). The Weinberg operator `(LᵀCεL)(HH)` gives a Majorana ν mass `m_ν=−c v²/Λ` after EWSB (`majorana_mass_matrix` → symmetric matrix → `diagonalize_takagi`) plus `ν̄νh`/`ν̄νhh` couplings (`extract_majorana_vertices` + `majorana_feynman_rule`). Key design points (pinned in `test_majorana.py`): (1) the atom **canonicalises its legs** at construction using the Grassmann symmetry `ψ₁ᵀMψ₂ = s·ψ₂ᵀMψ₁`, `s` from `majorana_symmetry_sign(Γ)` computed against an explicit Dirac-basis rep (`_dirac_rep`) — `s=+1` for `C P_L` ⟹ the mass matrix is symmetric; only `C·P_L`/`C·P_R` structures are supported (a Majorana *current* `Cγ^μP` raises — out of D.2 scope). (2) The gauge/discrete invariance transform works on `MajoranaBilinear` **for free** — `_fermion_transform`/`_fermion_transform_discrete` `.replace()` every `Indexed` leg tree-wide regardless of the wrapping Function; only the `has(Bilinear)` guards (invariance.py ×2 + the mass-dim `replace`, suggest.py ×2) were widened. Both legs being *field* components correctly get `+iαT`, and the ε-contraction makes `LᵀCεL HH` gauge-invariant. (3) `_eval_conjugate`: `(ψᵀCP_Lψ)† = ψ̄ C P_R ψ̄` (bar legs, projector flips, `C` invariant) — so `O + h.c.` is manifestly hermitian. (4) `suggest_yukawa(..., max_dim=5)` enumerates the Weinberg operator via a **same-chirality** `(F,F)` branch (`_majorana_leg_options`, the `(εFH)(εFH)` contraction); `max_dim=4` omits it. The same machinery handles a bare type-I-seesaw `½ M_R ν_Rᵀ C ν_R`. **UFO export of Majorana vertices is not yet supported** (MadGraph Majorana-fermion export is a separate follow-up). Worked model: `examples/sm_weinberg.py`.

**Type-I seesaw + physical couplings** (`vacuum/masses.py`: `seesaw_mass_matrix`/`seesaw_light_mass`; `vacuum/diagonalize.py`: `MajoranaRotation`): SM + 3 `ν_R` with Dirac Yukawa (`m_D=Y_ν v/√2` via `fermion_mass_matrix`) + Majorana mass (`M_R` via `majorana_mass_matrix`) → the symmetric `[[0,m_D],[m_Dᵀ,M_R]]` matrix → `diagonalize_takagi` gives the light (`m_ν≈−m_D M_R⁻¹ m_Dᵀ`, matches the exact Takagi light block) + heavy (`~M_R`) spectrum. The **physical neutrino couplings** need `MajoranaRotation`: the mass eigenstates are Majorana fields mixing `ν_L` with the charge-conjugate `ν_R^c`, so the square-only `Rotation` can't express it. With `diagonalize_takagi`'s `M=U D Uᵀ` convention the LH basis satisfies `n=U*ψ`, and since `ψ^c=ψ` ⟹ `(P_Lψ)^c=P_Rψ` the four Weyl legs substitute as `ν_L→U*·P_Lχ`, `ν̄_L→U·χ̄P_R`, `ν_R→U·P_Rχ`, `ν̄_R→U*·χ̄P_L` (a **rectangular** N→2N, conjugation/chirality-aware substitution — pass the ACTUAL `U` matrix, not a symbolic `IndexedBase`, whose `U[i,k]` would collide with the field leg in `expand_bilinear`). `MajoranaRotation.apply(expr, gen_indices, n_gen)` expands the flavour index over generations (U is numeric) and substitutes; extraction then gives `W ℓ̄ N=(g/√2)·U*[ℓ,heavy]` = the light–heavy mixing `V≈m_D/M_R`, → 0 as `M_R→∞` (decoupling). **Gotcha (fixed):** the leg-base match in `_leg` must use `==`, not `is` — a rotation's `xreplace` reconstructs the Lagrangian so its `IndexedBase` becomes a name-equal but object-distinct instance (SymPy doesn't intern `IndexedBase`), and an `is` check silently leaves the field un-rotated (couplings come out 0 on a rebuilt model). Worked model: `examples/sm_seesaw.py`; tutorial `examples/SM_Seesaw_Tutorial.ipynb`. NB: this surfaced+fixed a latent `check_mass_dimension` bug — collapsing every bilinear to the same `u³` erased leg identity, so the `(εH)(εH)` structure `H₀²−2H₀G⁺+G⁺²` spuriously cancelled to `0` under the shared-`u` substitution; the fix computes the dimension **per additive term**.

Because `Bilinear` and `PartialMu` are custom SymPy `Function`s with no `fdiff` rule, code that needs to differentiate an expression containing one of them **must never let the differentiation variable end up inside their arguments** — SymPy's chain rule falls back to an unevaluated, non-cancelling `Subs(Derivative(...))` placeholder otherwise. This has caused two real bugs (both fixed): gauge-invariance checking of `Dmu`-built kinetic terms, and of fermion `Bilinear` terms. The fix pattern in both cases: exploit that the transform commutes with the operator (∂_μ or the flavor index) and substitute *outside*, or explicitly redistribute the operator over `Add` args first (see `_transform_map`/`_fermion_transform`/`_expand_bilinear` in `invariance.py`, and `D_linear` in `operators.py`) — never call `sp.diff` on an expression where `PartialMu`/`Bilinear` wraps a variable-dependent argument.

Related gotcha: `Expr.as_coeff_Mul()` defaults to `rational=True` and only pulls out *Rational* coefficients, silently leaving Dummy/Symbol factors bundled into the "atom" side. Any code splitting `coeff * atom` terms (as the bilinear-distribution logic does) must find the atom explicitly via `as_ordered_factors()`, not rely on `as_coeff_Mul()`.

**Fermion mass-basis rotations** work through the ordinary `Rotation`/`Model.rotate` machinery with `Indexed` old-fields (e.g. `Rotation([eL[i], EL[i]], [e1L[i], e2L[i]], rotation_2x2(θ))`) — register **two rotations per chirality** (field-side AND bar-side, same matrix), and write every Lagrangian term with the same flavor-index symbol so the `xreplace` keys match. The resulting `Add`-valued legs inside `Bilinear` slots are distributed by `expand_bilinear` (`vertices/bilinear.py`, moved from invariance.py), which `extract_fermion_vertices` and `fermion_mass_matrix` apply automatically — without it, extraction silently grouped by unsplit composite keys. For symbolic 2×2 Dirac (biunitary/SVD) diagonalization use `diagonalize_svd_2x2` (θ_L from `M·Mᵀ`, θ_R from `Mᵀ·M`, analytic tan-2θ route) — `diagonalize_svd`'s generic `.diagonalize()` path returns unusable nested radicals for symbolic entries. `DiracFermion` is unsupported and **raises on construction** (a 4-component path would need its own physics-validated mass/diagonalization sector) — model vector-like fermions as two `WeylFermion`s with identical reps (see `examples/sm_vll.py`). Note `dirac_conjugate` strips a stray `diracI` factor (so a vector current `γ^μ·diracI` conjugates cleanly).

Fermion-bilinear **hermiticity** is checked via `Bilinear._eval_conjugate` (`vertices/bilinear.py`), which implements `(ψ̄₁Γψ₂)† = ψ̄₂Γ̄ψ₁` with `Γ̄ = γ⁰Γ†γ⁰` computed by `dirac.dirac_conjugate` (covers `diracI`, `diracPL`/`diracPR`, bare `DiracGamma(mu)`, and `DiracGamma(mu)*diracPL`/`diracPR` — raises `NotImplementedError` for anything else, e.g. products of ≥2 gamma matrices, rather than guessing) and a `bar_partner` registry (`fields.py`) that every `Fermion` populates at construction to find "the bar of this field"/"the field of this bar". A Yukawa term written without its `+ h.c.` partner now genuinely fails `Model.check_invariance()`'s hermiticity check; a gauge current (`fermion_gauge_current`) is hermitian by construction with no hand-written h.c. needed.

**Discrete-symmetry invariance for fermion multiplets** (`ZN`/`S3` acting on fermion generations — e.g. two lepton doublets forming an S₃ doublet, matching `thdm_s3.py`'s scalar S₃ pattern but for leptons) is also checked, via `DiscreteSymmetry.fermion_generator_data()` (`groups/discrete.py`) + `_fermion_transform_discrete` (`invariance.py`). Unlike the gauge case, discrete transformations are *finite* (no `α`-linearization/differentiation needed) — a multiplet member substitutes directly as `ψ'_i = Σ_k M[i,k]ψ_k`. The bar leg uses `X = (M⁻¹)ᵀ`, **not** `M` itself (derived from requiring `Σ_i ψ̄'_iψ'_i` invariant, verified by direct computation, not assumed) — `X` only coincides with `M` when `M` is real orthogonal (true for `S3`'s irreps, not `ZN`'s complex-phase ones, where `X = M̄`). `assign()` rejects multiplets mixing `Fermion` and non-`Fermion` members (they'd need different substitution mechanisms — flat dict vs. index-preserving `.replace()`).

### Model.validate() umbrella and the charge/hermiticity vertex checks

`Model.validate()` (`lagrangian.py`) aggregates every applicable check into one `ValidationReport`: `check_invariance`, `check_anomalies` (`anomalies.py`, skipped without charged fermions), the UFO round-trip (`verify/ufo_roundtrip.py::verify_ufo_numeric`, only with a `ufo_path`), and — when a declared `charges` map + physical `fields` are passed — the three `charges.py` checks. **Electric charge is never hardcoded as `Q=T3+Y`**: `derive_charge_operator` solves for the diagonal-generator combination annihilating every VEV (null-space; excludes groups no VEV'd scalar transforms under, so colour/spectator U(1)s drop out; raises on rank-0 "charge-breaking vacuum" and rank-2 "ambiguous" cases), and `physical_charges` propagates weak→physical charges through the registered `Rotation`s as `Q_new = R·Q_old·R⁻¹` (this is what makes `W1/W2 → W±` come out ±1 automatically). The declared map is the authority for the vertex checks; the derivation is the cross-check (`check_charge_consistency`, scale fixed by the *most common* declared/derived ratio so a single mistuned declaration is flagged, not the anchor). `check_hermiticity_pairing` needs the partner coupling `(-1)^{1+d}·conj(coupling)` where **d = momentum-tag degree** (0 → −conj, 1 → +conj: the VSS acid test — derived empirically, pinned in `test_charges.py`), and requires an explicit full `conjugates` antiparticle map because charged physical fields like `W±` come from rotations, not `conjugate_pair` (so they're absent from `conjugate_map`).

### Quark flavour / CKM (`flavor.py`)

CKM is done the FeynRules-SM way, **never** a symbolic 3×3 Yukawa SVD (generic complex Yukawas don't diagonalize in closed form — nested radicals): work in the quark **mass basis** with diagonal Yukawas and **insert** V into the charged current only (`g/√2 W⁺ ū_i γ^μP_L V_ij d_j`), leaving neutral currents flavour-diagonal (GIM). `standard_ckm()` builds V from the exact PDG standard parametrization (3 angle externals + 1 phase + 9 complex-internal `V_ij`), unitary as a trig identity so `V†V` simplifies to `1` (a matrix of 9 independent symbols is NOT unitary and would fake Z-FCNCs). CKM elements must be **scalar** parameters (`ParameterSet` is scalar-only; the UFO printer emits an invalid `V[0,1]` subscript for an `IndexedBase`); the 9 `V_ij` export as `type='complex'` internals and round-trip. The 2-gen Cabibbo case IS derivable through the ordinary real `Rotation`/`rotation_2x2` machinery — `test_ckm.py::TestTwoGenerationGIM` pins the GIM cancellation (mixing in W, exact `cos²+sin²` diagonality in Z). Worked demo: `examples/sm_ckm.py`.

### Reusable SM builders (`models.py`)

The electroweak scaffold (SU(2)×U(1) groups, Higgs doublet + potential + kinetic, and the physical-basis rotations) is identical across every `examples/sm_*.py`, so `models.py` factors it out (precedent: `flavor.py::standard_ckm` — the library ships the *standard* construction). **Primitives**: `electroweak_gauge()` → `(SU2L,U1Y,gw,g1)`; `higgs_doublet(SU2L,U1Y, v=, mh=, lam=, extra_reps=)` → Higgs with VEV expanded (`lam` overrides the `mh²/2v²` default; `extra_reps` merges e.g. a U(1)_X charge); `weinberg_rotation(model, SU2L, U1Y, z="Z")` (`z` overridable — u1x passes `"Z0"` for its chained Z–Z′) and `charged_current_rotation(model, SU2L, wp=, wm=)`. **Bundles**: `electroweak_scaffold(...)` → `ElectroweakScaffold` (`.gauge_groups`, `.parameters`, `.fields`, `.add_higgs(L)`); `to_physical_basis(model, scaffold)` runs both rotations + wires the Goldstone `h/G0/Gp/Gm`+`cmap`, returning `PhysicalBasis` (`.bosons` = `[h,G0,Gp,Gm,Z,A,Wp,Wm]`); `standard_model(generations=, quarks=, physical_basis=)` → `StandardModel` for the vanilla case. **No `pheno` dependency** (Yukawas/`DiracParticle`s stay in the decay examples). Key design point: the Higgs helper contributes terms to the *caller's* `Lagrangian` (via `add_higgs`) rather than sealing a Model, because extension files assemble the Model with their own extra fields/params. Full reuse: `sm_decays`/`sm_higgs_decays`/`sm_vll`/`sm_seesaw`; partial (own quark/QCD or chained rotations): `sm_scalar_gauge`/`sm_u1x`. `tests/test_models.py` pins `m_Z²=(g²+g'²)v²/4`, `m_W²=g²v²/4`, massless photon, `m_h²=2λv²`, invariance, the u1x-style chained `weinberg_rotation(z="Z0")`, and that the refactored `sm_decays` still yields `Γ(Z→νν̄)=m_Z(g²+g'²)/96π` (behaviour-preserving). Notebooks: Particle_Decays §4 uses the scaffold; SM_Seesaw/SM_VLL dedupe just the rotation block (their param declarations stay inline — pedagogical, some `tex=`-annotated). SM_Feynman_Rules stays fully inline (its lesson *is* the build).

### Decay widths (`pheno/`) — Feynman rules → observables

Tree-level **1→2 partial widths, total widths and branching ratios** from a built `Model` (`Γ = ⟨|M|²⟩·√λ(M²,m₁²,m₂²)/(16πM³)`). The `|M|²` path is deliberately **covariant, not explicit 4×4 matrices**: Dirac traces go through `sympy.physics.hep.gamma_matrices` (`gamma_trace`/`kahane_simplify` — ships with SymPy, so still pure-SymPy, no new dependency), contracted with spin/polarization sums and reduced to on-shell `p_i·p_j` only at the end. Both routes were prototyped and give identical widths; covariant was chosen for frame-independent `|M|²` and a cleaner path to a later 2→2. **The explicit-matrix evaluator lives on as the test oracle** (`tests/test_pheno.py::_oracle_*`), sharing no code with the engine, so no width rests on the covariant path alone.

Three gaps in SymPy's engine, all filled in `pheno/lorentz.py` (each found empirically, not guessed): (1) `gamma_trace` **crashes on mass terms** — `p̸+m` is a `TensAdd` whose scalar arg has no `.sorted_components()`; `dirac_trace` splits scalar pieces off and applies `Tr[c·I₄]=4c`. (2) **No γ₅**, so `P_L`/`P_R` are unrepresentable; `reduce_projectors` pushes projectors right (`P_{L,R}γ^μ=γ^μP_{R,L}`, `P²=P`, `P_LP_R=0`) and uses `Tr[X P]=½Tr[X]∓½Tr[Xγ₅]`, **dropping the ε term only where it provably vanishes** — a 1→2 final state has just 2 independent momenta and every free index meets a *symmetric* polarization sum, while ε is antisymmetric. This is a **hard guard**, not a comment: `reduce_projectors` takes `n_momenta`/`n_free_indices` and raises `NotImplementedError` outside that regime, so a future 1→3 fails loudly instead of returning a plausible wrong number (pinned in `test_pheno.py`). (3) `replace_with_arrays` **fails on contracted dummies** (SymPy 1.14, `ValueError: p1(L_0) not found`); `contract_to_dots` walks `TensMul.components`/`.get_indices()`/`.coeff` directly — the one place touching semi-private tensor API, isolated on purpose and pinned by tests.

Two gotchas that produce *silently wrong numbers*: (1) **a Dirac fermion is two `WeylFermion`s**, so `e_L` and `e_R` are different legs and `extract_fermion_vertices` reports the L and R currents under different keys — `DecayCalculator(particle_map={eL[i]: e, eR[i]: e, ...})` merges them; omitting it yields two half-channels each missing the other chirality *and* the `g_Lg_R` mass interference. (2) **thresholds**: with symbolic masses `M ≥ m₁+m₂` is undecidable so the channel is kept (correct — the user may substitute masses that open it), but substituting numbers into a closed channel makes `√λ` **imaginary** and poisons the total width and every BR (`h→WW` at `m_h=125` is exactly this) — use `numeric_partial_widths`/`numeric_branching_ratios`, which re-test the threshold after substitution, **not** `numeric(partial_widths(...))`.

`pheno/vertices.py`'s `collect_decay_vertices` unifies the two extraction tracks into one `DecayVertex` list, doing the `P_L`/`P_R` split **once** (every existing caller — `scripts/export_sm_ufo.py`, `examples/sm_u1x.py`, `examples/sm_vll.py` — hand-rolls it); a candidate for promotion into core. `pheno/trace.py` is a separate small direct evaluator (≤4 γ) kept as an *independent* cross-check of the conventions, not on the compute path — building it out surfaced and fixed a real bug there: `γ^μγ^μ` collapses to a `Pow`, which its `Mul`-only dispatch missed, returning `4·gamma(0)**2` instead of `4g^{μμ}`. Implemented vertex types: **SSS, FFS, FFV, VVS**; `VSS`/`VVV` carry `p(φ)` momentum tags from `to_momentum_space` and raise (derivative couplings are not yet squared). Worked model: `examples/sm_decays.py` (reproduces `Γ(Z→νν̄)=0.168`, `Γ(Z→ττ)=0.084`, `Γ(W→τν)=0.227` GeV, `Γ(h→ττ)=0.26` MeV vs PDG).

`docs/manual/decays_roadmap.md` (chapter 16) is the tiered plan for the full Higgs BR picture: Tier 1 (tree `f̄f`, small — the engine already computes it) recommends a post-EWSB **`DiracParticle`** abstraction replacing the `particle_map`/`masses`/`color_factors` dict triple — note an *input-level* `DiracFermion` stays impossible (SM fermions are chiral: `b_L` doublet `Y=+1/6` vs `b_R` singlet `Y=−1/3`, one rep slot per field; only the vector-like equal-reps case could get it as sugar); Tier 2 (off-shell `VV*`: propagators + 1→3 numeric phase space + diagram assembly, large); Tier 3 (loop-induced `gg`/`γγ` via effective `A_{1/2}`/`A_1` form-factor vertices, a documented ethos exception with the CKM shortcut as precedent).

**Tier 1 DiracParticle: shipped** (`pheno/particles.py`). `DiracParticle(name, left, right, mass, color=1)` bundles the two Weyl leg *bases* (bar legs auto-derived via `bar_partner`), the antiparticle, mass and `N_c`; `DecayCalculator(model, {bosons:…}, particles=[…])` consumes a list. `expand_particles` merges them into the calc's internal `particle_map`/`masses`/colour registry, and colour is applied **once per channel** in `_channel_color` (the per-leg `∏ _color_of` legacy path stays for `color_factors=`). This makes the three traps unrepresentable and adds `calc.unmatched_channels` (a list of FFS/FFV vertices claimed by no declared particle, with a one-shot `warnings.warn`) so a forgotten fermion is visible, not silently dropped. Backward-compatible: the old `particle_map`/`masses`/`color_factors` triple is untouched and composes with `particles=` (e.g. a LH-only neutrino via `particle_map`). Worked model `examples/sm_higgs_decays.py` (b̄b-dominated ~84% with running masses); tutorial §10 adds a full-fermion `DiracParticle` section + BR bar chart. **Colour over-count trap (verified, now only reachable via the low-level path):** a decay-facing quark channel needs ONE colour component + colour applied ONCE → `3y²`; the old footgun (Yukawa colour sum AND `particle_map` collapse of the 3 components AND `color_factors` on both legs) compounds coherent-coupling 9× with 3×3 → `81y²`, a silent 27× over-count — `DiracParticle`'s colour-once semantics prevents it.

### MadGraph round-trip (the exported UFO actually computes right)

`scripts/export_sm_ufo.py` + `scripts/madgraph_roundtrip.py` export an EW+leptons SM UFO and compare cross sections to MadGraph's stock `sm` at a matched parameter point (`e+e-→μ+μ- = 2.788 pb`, `e+e-→W+W- = 19.50 pb`, both to MC precision; `docs/benchmark.md`). Not in CI (each launch compiles Fortran). MG5 lives at `~/.local/mg5dl/MG5_aMC_v3_7_2` and runs on Python 3.13. **Three UFO-export issues this surfaced** (first two fixed in the library + regression-tested, invisible to unit tests but fatal in MG): (1) generated UFO files must use **absolute** imports (`from object_library import …`, not `from .object_library`) — MG runs them as standalone scripts for param-card generation; `verify/ufo_roundtrip.py` + the test `_import_ufo` helpers now load a UFO the MG way (dir on `sys.path`, import submodules by bare name). (2) `add_fermion_vertex` must apply the **Feynman-rule `i`** to the L/R couplings (the bosonic/VVV paths already carry it) — omitting it is invisible in `e+e-→μμ` (overall phase cancels in |M|²) but breaks FFV↔VVV interference in `e+e-→WW` (gives 98 pb not 19.5). (3) The triple-gauge coupling from the **complex W± rotation** comes out of `cubic_couplings` with the opposite overall sign to MG's VVV1 convention — the *export script* flips `gAWW`/`gZWW` (validated by the WW round-trip); the real-basis `ggg=-g_s` is unaffected and already matches stock, so this is NOT a universal `add_vvv_vertex` negation.

### SU(3) / QCD vertex dynamics

The group theory (Gell-Mann generators/structure constants, `groups/gauge.py` + `groups/sun.py`) and the vertex-extraction machinery (`Dmu`, `fermion_gauge_current`, `cubic_couplings`/`quartic_couplings`) are fully generic over gauge group — no SU(3)-specific code, and since generators are built dynamically (see the Declare-stage note above) the machinery works unchanged for any SU(N) irrep (`yangmills._fundamental_label` now reads `group.N` instead of an `isinstance(SU2/SU3)` dispatch; `structure_constants` delegates to the group's own cached `f`; `anomalies._cubic_index` computes `A(R)` from the symmetric trace normalized to the fundamental, so any complex rep — not just the SU(3) triplet — is handled, with `A(R̄)=−A(R)` free from the conjugate generators). **Gotcha (fixed):** `Model.check_invariance()` on a `Dmu`-built kinetic term of a large/radical-valued rep (a general GT irrep) used to expand the full multi-generator variation at once, which is intractable for such reps; `invariance.gauge_variation` now checks each generator's Ward identity from an *independent single-α* transformation (mathematically identical — the cross-generator O(α²) terms vanish at α=0 anyway — but keeps every intermediate expression small, and is never slower for the SM-scale reps). It is still symbolic and thus not instantaneous for big reps (a sextet/SU(4)-fundamental kinetic term is tens of seconds); mass terms and the direct algebra/anomaly pins are the fast route (`test_sun.py`). What closed the gap: `examples/sm_scalar_gauge.py` now includes a quark sector (SU(2) doublet `QL` + `SU3c`-triplet singlets `uR`/`dR`, flavor-generic `Yu`/`Yd` Yukawas mirroring the lepton pattern — 3 generations, undiagonalized, no CKM: CKM mixing is physically orthogonal to SU(3) vertex dynamics and a fully generic 3-generation complex-Yukawa SVD doesn't resolve in closed symbolic form) and a gluon field; `export/ufo/vvvv.py`'s `assemble_vvvv` builds the 3-structure VVVV1/2/3 decomposition of the 4-boson self-coupling that `yangmills.py`'s `quartic_couplings()` docstring had deferred as unbuilt "Phase 5" (derived by direct functional differentiation of `-¼F^aF^a`, cross-checked against both SU(2) and SU(3) in `tests/test_yangmills.py`); `export/ufo/writer.py` emits real color-tensor strings (`T(3,1,2)` for qqg, `f(1,2,3)` for ggg, three `f*f` structures for gggg) instead of the hardcoded singlet `'1'` — gluons are exported as **one** UFO particle repeated (color summed via the tensor string), never as 8 separate weak-basis components.

### Conventions (all pinned by tests — see `CONVENTIONS.md` for the full list)

Metric `(+,−,−,−)`; `P_L=(1−γ₅)/2`; `D_μ=∂_μ−igT^aA^a_μ`; VEV expansion with explicit `1/√2`; Feynman rule = `i × coefficient × ∏(field multiplicity)!`; momentum convention `∂_μφ→ip(φ)φ`; simplification hierarchy `expand→collect→factor→simplify` (simplify last); positive-assumption dummies for anything under `sqrt`; every physical result gets dual verification (symbolic diff **and** `verify.numeric_equal` random-point check).

The notebooks are tracked through a `nbstripout` git filter (`.gitattributes`, `*.ipynb filter=nbstripout diff=ipynb`), configured with `--keep-output` so real outputs (plots, printed Feynman rules) stay in git history and in diffs — only `execution_count` and volatile per-cell metadata get normalized away on `git add`/`git diff`, so re-executing a notebook without changing its actual results produces no diff noise. A fresh clone needs to activate the filter once (`.gitattributes` alone only declares the pattern, it doesn't install the filter driver):

```bash
nbstripout --install --attributes .gitattributes
git config filter.nbstripout.clean "$(git config filter.nbstripout.clean) --keep-output"
git config diff.ipynb.textconv "$(git config diff.ipynb.textconv) --keep-output"
```

(`nbstripout --install` itself does not persist CLI flags like `--keep-output` into the stored filter command — they have to be appended to `filter.nbstripout.clean`/`diff.ipynb.textconv` by hand, as above.)
