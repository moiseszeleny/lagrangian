# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

`feynlag` вЂ” a pure-SymPy library that derives tree-level Feynman rules from user-written BSM (Beyond Standard Model) Lagrangians. Input style is FeynRules-like: declare particle fields with gauge/discrete representations, write Lagrangian terms explicitly with library building blocks (`Dmu`, `dag`, `Bilinear`), and the library checks invariance, breaks the symmetry, diagonalizes, and extracts vertices.

## Architecture

### Pipeline stage в†’ module map

The library is organized around one linear physics pipeline; each stage lives in its own module and the top-level `Model` (in `lagrangian.py`) orchestrates all of them **lazily** (`functools`-style caching via `Model._cache`, invalidated on `solve_tadpoles`/`rotate` вЂ” nothing is computed at construction or at import time, deliberately, to avoid a known flaw in the DLRSM1 reference implementation this library was built from):

1. **Declare** вЂ” `parameters.py` (`ExternalParameter` fixed by experiment, `InternalParameter` derived later, `ParameterSet` topologically sorts internal dependencies), `fields.py` (`Scalar`/`Fermion`/`GaugeBoson`, auto-generated component symbols + conjugate registry), `groups/` (`gauge.py`: U1/SU2/SU3 with explicit generator matrices; `discrete.py`: ZN/S3 with ClebschвЂ“Gordan products).
2. **Write** вЂ” `Lagrangian`/`LagrangianTerm` (`lagrangian.py`) collect sector-tagged terms (`kinetic`, `potential`, `yukawa`, `gauge`, `other`); `operators.py` provides `Dmu` (covariant derivative) and `PartialMu` (the `в€‚_Ој` placeholder, resolved to momentum space later).
3. **Check** вЂ” `invariance.py`: gauge invariance by infinitesimal component transformation (`ОґП† = iО±^aT^aП†`), discrete invariance by finite generator substitution, hermiticity, mass-dimension power counting. `Model.check_invariance()` is the entry point.
4. **Break the symmetry** вЂ” `vacuum/ewsb.py` (`Vacuum`: VEV shift maps `П†вЃ°в†’(v+h+ia)/в€љ2`), `vacuum/tadpoles.py` (extract + solve, registers solutions on `InternalParameter`s), `vacuum/masses.py` (mass matrices: real, complex/charged via Dummy-conjugate trick, gauge-boson).
5. **Diagonalize** вЂ” `vacuum/diagonalize.py`: `Rotation` (weakв†”physical substitution + verification), 2Г—2 analytic (`tan 2Оё`), SVD (Dirac), Takagi (Majorana).
6. **Extract vertices** вЂ” `vertices/extract.py` (Poly-based commuting-symbol extractor, bosons only), `vertices/bilinear.py` (`Bilinear` вЂ” the fermion-sandwich track, separate from the extractor since fermions aren't commuting symbols), `vertices/yangmills.py` (cubic/quartic pure-gauge self-couplings from structure constants, not from the extractor), `vertices/vertex.py` (`Vertex` objects classified into the closed catalog SSS/SSSS/VSS/VVS/VVSS/VVV/VVVV/FFS/FFV).
7. **Export** вЂ” `export/latex.py`, `export/ufo/` (writer + UFO `object_library`/`lorentz_map`/`pycode` for MadGraph-importable model directories; `vvvv.py` assembles the 4-boson self-coupling into the 3 UFO Lorentz structures, and colored (SU(3)) vertices get real `T`/`f` color-tensor strings вЂ” see writer.py's `add_vvv_vertex`/`add_vvvv_vertex`/`add_fermion_vertex`).

`dirac.py` (Clifford algebra, chiral projectors `diracPL`/`diracPR`) and `verify/checks.py` (dual symbolic+numeric verification utilities, `numeric_equal`) are used across stages.

### The two-track extraction design (important, easy to get wrong)

Bosonic fields (scalars, gauge bosons) are represented as plain commuting `sympy.Symbol`s and go through the `Poly`-based extractor in `vertices/extract.py`. Fermion fields are `IndexedBase`-typed (one flavor-indexed component per generator), and a fermion sandwich `П€М„О“П€` is wrapped in the opaque `Bilinear(bar_indexed, gamma, field_indexed)` atom вЂ” this **never** enters the commuting-symbol extractor. `vertices/bilinear.py`'s `extract_fermion_vertices` groups terms by `(bar, gamma, field)` and peels boson legs off the coefficient with the bosonic extractor.

**Four-fermion (FFFF) operators** (dim-6 effective operators like the Fermi muon-decay operator `(П€М„О“П€)(П‡М„О“вЂІП‡)`) are the **two-bilinear** case of the same track (`extract_fermion_vertices` accepts one *or* two `Bilinear` factors; a nested `((bar,О“,field),(bar',О“вЂІ,field'))` sorted-pair key, distinguished from the flat one-bilinear key by `isinstance(key[0], tuple)`). Two design commitments, both pinned in `test_four_fermion.py`: (1) the **as-written bilinear basis** вЂ” no Fierz canonicalisation, a Fierz-rearranged operator is a *different input* (matches SMEFT-UFO practice); (2) **four distinct fermion components only** вЂ” a repeated component (incl. a squared bilinear `BВІ`, which `atoms()` would dedup, so `_bilinear_factors` counts multiplicity by walking factors) raises `NotImplementedError`, because identical legs generate cross-chain Wick contractions whose relative signs need genuine spinor-index Fierz algebra the opaque-`Bilinear` design cannot express. With distinct legs there are no exchange contractions, so the rule is the plain scalar `iВ·coeffВ·в€Џ(boson mult)!` (`four_fermion_feynman_rule`) with the two Dirac structures `(О“,О“вЂІ)` carried **separately** вЂ” folding them into one product would wrongly invoke the Clifford algebra *across* chains (contract a shared `Оі^Ој`). A vector-current contraction `(П€М„Оі^ОјP_LП€)(П‡М„Оі_ОјP_LП‡)` is written with a **shared index symbol**, `DiracGamma(mu)` on one leg and `DiracGammaLower(mu)` on the other (`dirac_conjugate` handles the lowered `Оі_Ој` вЂ” self-conjugate like `Оі^Ој` since the metric is real). The mass-dimension check gains a `max_dim` flag (`Model.check_invariance(max_dim=6)` / `validate(max_dim=6)`) to admit dim-6 operators; with the Wilson coefficient carrying its physical dim в€’2 the *term* is still dim-4, so the flag matters only under the operator-dimension convention (dimensionless coefficient). UFO export: 8 `FFFF*` Lorentz structures (SвЉ—S / VвЉ—V with a contracted Lorentz dummy) in `lorentz_map.py`, `writer.add_four_fermion_vertex` (carries the Feynman-rule `i`), `write_ufo(four_fermion_vertices=вЂ¦)`. **Export the h.c. vertex too**: a Hermitian `op + h.c.` is two FFFF vertices, and MadGraph needs the conjugate pair to route fermion-number flow through a contact interaction (a UFO with only one в†’ `NoDiagramException`; this is a MadGraph requirement, not a feynlag one вЂ” the exported Lorentz string is verbatim the one MadGraph's shipped `taudecay_UFO` uses). Validated end-to-end: `scripts/madgraph_fermi.py` reproduces the muon width `О“(Ојв†’eОЅОЅ)=G_FВІm_ОјвЃµ/192ПЂВі` to 0.07% (`docs/benchmark.md`, not in CI). Worked model: `examples/fermi_theory.py`.

**Majorana bilinears / the dim-5 Weinberg operator** (`MajoranaBilinear` in `vertices/bilinear.py`) are the same-chirality, charge-conjugation-contracted sandwich `П€в‚ЃбµЂ C О“ П€в‚‚` вЂ” **both legs are field components** (no bar), the middle `О“` **must** contain the charge-conjugation matrix `diracC` (`dirac.py`, `C=iОіВІОівЃ°`, with `CбµЂ=CвЂ =CвЃ»В№=в€’C`). The Weinberg operator `(LбµЂCОµL)(HH)` gives a Majorana ОЅ mass `m_ОЅ=в€’c vВІ/О›` after EWSB (`majorana_mass_matrix` в†’ symmetric matrix в†’ `diagonalize_takagi`) plus `ОЅМ„ОЅh`/`ОЅМ„ОЅhh` couplings (`extract_majorana_vertices` + `majorana_feynman_rule`). Key design points (pinned in `test_majorana.py`): (1) the atom **canonicalises its legs** at construction using the Grassmann symmetry `П€в‚ЃбµЂMП€в‚‚ = sВ·П€в‚‚бµЂMП€в‚Ѓ`, `s` from `majorana_symmetry_sign(О“)` computed against an explicit Dirac-basis rep (`_dirac_rep`) вЂ” `s=+1` for `C P_L` вџ№ the mass matrix is symmetric; only `CВ·P_L`/`CВ·P_R` structures are supported (a Majorana *current* `CОі^ОјP` raises вЂ” out of D.2 scope). (2) The gauge/discrete invariance transform works on `MajoranaBilinear` **for free** вЂ” `_fermion_transform`/`_fermion_transform_discrete` `.replace()` every `Indexed` leg tree-wide regardless of the wrapping Function; only the `has(Bilinear)` guards (invariance.py Г—2 + the mass-dim `replace`, suggest.py Г—2) were widened. Both legs being *field* components correctly get `+iО±T`, and the Оµ-contraction makes `LбµЂCОµL HH` gauge-invariant. (3) `_eval_conjugate`: `(П€бµЂCP_LП€)вЂ  = П€М„ C P_R П€М„` (bar legs, projector flips, `C` invariant) вЂ” so `O + h.c.` is manifestly hermitian. (4) `suggest_yukawa(..., max_dim=5)` enumerates the Weinberg operator via a **same-chirality** `(F,F)` branch (`_majorana_leg_options`, the `(ОµFH)(ОµFH)` contraction); `max_dim=4` omits it. The same machinery handles a bare type-I-seesaw `ВЅ M_R ОЅ_RбµЂ C ОЅ_R`. **UFO export of Majorana vertices is not yet supported** (MadGraph Majorana-fermion export is a separate follow-up). Worked model: `examples/sm_weinberg.py`.

**Type-I seesaw + physical couplings** (`vacuum/masses.py`: `seesaw_mass_matrix`/`seesaw_light_mass`; `vacuum/diagonalize.py`: `MajoranaRotation`): SM + 3 `ОЅ_R` with Dirac Yukawa (`m_D=Y_ОЅ v/в€љ2` via `fermion_mass_matrix`) + Majorana mass (`M_R` via `majorana_mass_matrix`) в†’ the symmetric `[[0,m_D],[m_DбµЂ,M_R]]` matrix в†’ `diagonalize_takagi` gives the light (`m_ОЅв‰€в€’m_D M_RвЃ»В№ m_DбµЂ`, matches the exact Takagi light block) + heavy (`~M_R`) spectrum. The **physical neutrino couplings** need `MajoranaRotation`: the mass eigenstates are Majorana fields mixing `ОЅ_L` with the charge-conjugate `ОЅ_R^c`, so the square-only `Rotation` can't express it. With `diagonalize_takagi`'s `M=U D UбµЂ` convention the LH basis satisfies `n=U*П€`, and since `П€^c=П€` вџ№ `(P_LП€)^c=P_RП€` the four Weyl legs substitute as `ОЅ_Lв†’U*В·P_LП‡`, `ОЅМ„_Lв†’UВ·П‡М„P_R`, `ОЅ_Rв†’UВ·P_RП‡`, `ОЅМ„_Rв†’U*В·П‡М„P_L` (a **rectangular** Nв†’2N, conjugation/chirality-aware substitution вЂ” pass the ACTUAL `U` matrix, not a symbolic `IndexedBase`, whose `U[i,k]` would collide with the field leg in `expand_bilinear`). `MajoranaRotation.apply(expr, gen_indices, n_gen)` expands the flavour index over generations (U is numeric) and substitutes; extraction then gives `W в„“М„ N=(g/в€љ2)В·U*[в„“,heavy]` = the lightвЂ“heavy mixing `Vв‰€m_D/M_R`, в†’ 0 as `M_Rв†’в€ћ` (decoupling). **Gotcha (fixed):** the leg-base match in `_leg` must use `==`, not `is` вЂ” a rotation's `xreplace` reconstructs the Lagrangian so its `IndexedBase` becomes a name-equal but object-distinct instance (SymPy doesn't intern `IndexedBase`), and an `is` check silently leaves the field un-rotated (couplings come out 0 on a rebuilt model). Worked model: `examples/sm_seesaw.py`; tutorial `examples/SM_Seesaw_Tutorial.ipynb`. NB: this surfaced+fixed a latent `check_mass_dimension` bug вЂ” collapsing every bilinear to the same `uВі` erased leg identity, so the `(ОµH)(ОµH)` structure `Hв‚ЂВІв€’2Hв‚ЂGвЃє+GвЃєВІ` spuriously cancelled to `0` under the shared-`u` substitution; the fix computes the dimension **per additive term**.

Because `Bilinear` and `PartialMu` are custom SymPy `Function`s with no `fdiff` rule, code that needs to differentiate an expression containing one of them **must never let the differentiation variable end up inside their arguments** вЂ” SymPy's chain rule falls back to an unevaluated, non-cancelling `Subs(Derivative(...))` placeholder otherwise. This has caused two real bugs (both fixed): gauge-invariance checking of `Dmu`-built kinetic terms, and of fermion `Bilinear` terms. The fix pattern in both cases: exploit that the transform commutes with the operator (в€‚_Ој or the flavor index) and substitute *outside*, or explicitly redistribute the operator over `Add` args first (see `_transform_map`/`_fermion_transform`/`_expand_bilinear` in `invariance.py`, and `D_linear` in `operators.py`) вЂ” never call `sp.diff` on an expression where `PartialMu`/`Bilinear` wraps a variable-dependent argument.

Related gotcha: `Expr.as_coeff_Mul()` defaults to `rational=True` and only pulls out *Rational* coefficients, silently leaving Dummy/Symbol factors bundled into the "atom" side. Any code splitting `coeff * atom` terms (as the bilinear-distribution logic does) must find the atom explicitly via `as_ordered_factors()`, not rely on `as_coeff_Mul()`.

**Fermion mass-basis rotations** work through the ordinary `Rotation`/`Model.rotate` machinery with `Indexed` old-fields (e.g. `Rotation([eL[i], EL[i]], [e1L[i], e2L[i]], rotation_2x2(Оё))`) вЂ” register **two rotations per chirality** (field-side AND bar-side, same matrix), and write every Lagrangian term with the same flavor-index symbol so the `xreplace` keys match. The resulting `Add`-valued legs inside `Bilinear` slots are distributed by `expand_bilinear` (`vertices/bilinear.py`, moved from invariance.py), which `extract_fermion_vertices` and `fermion_mass_matrix` apply automatically вЂ” without it, extraction silently grouped by unsplit composite keys. For symbolic 2Г—2 Dirac (biunitary/SVD) diagonalization use `diagonalize_svd_2x2` (Оё_L from `MВ·MбµЂ`, Оё_R from `MбµЂВ·M`, analytic tan-2Оё route) вЂ” `diagonalize_svd`'s generic `.diagonalize()` path returns unusable nested radicals for symbolic entries. `DiracFermion` is unsupported and **raises on construction** (a 4-component path would need its own physics-validated mass/diagonalization sector) вЂ” model vector-like fermions as two `WeylFermion`s with identical reps (see `examples/sm_vll.py`). Note `dirac_conjugate` strips a stray `diracI` factor (so a vector current `Оі^ОјВ·diracI` conjugates cleanly).

Fermion-bilinear **hermiticity** is checked via `Bilinear._eval_conjugate` (`vertices/bilinear.py`), which implements `(П€М„в‚ЃО“П€в‚‚)вЂ  = П€М„в‚‚О“М„П€в‚Ѓ` with `О“М„ = ОівЃ°О“вЂ ОівЃ°` computed by `dirac.dirac_conjugate` (covers `diracI`, `diracPL`/`diracPR`, bare `DiracGamma(mu)`, and `DiracGamma(mu)*diracPL`/`diracPR` вЂ” raises `NotImplementedError` for anything else, e.g. products of в‰Ґ2 gamma matrices, rather than guessing) and a `bar_partner` registry (`fields.py`) that every `Fermion` populates at construction to find "the bar of this field"/"the field of this bar". A Yukawa term written without its `+ h.c.` partner now genuinely fails `Model.check_invariance()`'s hermiticity check; a gauge current (`fermion_gauge_current`) is hermitian by construction with no hand-written h.c. needed.

**Discrete-symmetry invariance for fermion multiplets** (`ZN`/`S3` acting on fermion generations вЂ” e.g. two lepton doublets forming an Sв‚ѓ doublet, matching `thdm_s3.py`'s scalar Sв‚ѓ pattern but for leptons) is also checked, via `DiscreteSymmetry.fermion_generator_data()` (`groups/discrete.py`) + `_fermion_transform_discrete` (`invariance.py`). Unlike the gauge case, discrete transformations are *finite* (no `О±`-linearization/differentiation needed) вЂ” a multiplet member substitutes directly as `П€'_i = ОЈ_k M[i,k]П€_k`. The bar leg uses `X = (MвЃ»В№)бµЂ`, **not** `M` itself (derived from requiring `ОЈ_i П€М„'_iП€'_i` invariant, verified by direct computation, not assumed) вЂ” `X` only coincides with `M` when `M` is real orthogonal (true for `S3`'s irreps, not `ZN`'s complex-phase ones, where `X = MМ„`). `assign()` rejects multiplets mixing `Fermion` and non-`Fermion` members (they'd need different substitution mechanisms вЂ” flat dict vs. index-preserving `.replace()`).

### Model.validate() umbrella and the charge/hermiticity vertex checks

`Model.validate()` (`lagrangian.py`) aggregates every applicable check into one `ValidationReport`: `check_invariance`, `check_anomalies` (`anomalies.py`, skipped without charged fermions), the UFO round-trip (`verify/ufo_roundtrip.py::verify_ufo_numeric`, only with a `ufo_path`), and вЂ” when a declared `charges` map + physical `fields` are passed вЂ” the three `charges.py` checks. **Electric charge is never hardcoded as `Q=T3+Y`**: `derive_charge_operator` solves for the diagonal-generator combination annihilating every VEV (null-space; excludes groups no VEV'd scalar transforms under, so colour/spectator U(1)s drop out; raises on rank-0 "charge-breaking vacuum" and rank-2 "ambiguous" cases), and `physical_charges` propagates weakв†’physical charges through the registered `Rotation`s as `Q_new = RВ·Q_oldВ·RвЃ»В№` (this is what makes `W1/W2 в†’ WВ±` come out В±1 automatically). The declared map is the authority for the vertex checks; the derivation is the cross-check (`check_charge_consistency`, scale fixed by the *most common* declared/derived ratio so a single mistuned declaration is flagged, not the anchor). `check_hermiticity_pairing` needs the partner coupling `(-1)^{1+d}В·conj(coupling)` where **d = momentum-tag degree** (0 в†’ в€’conj, 1 в†’ +conj: the VSS acid test вЂ” derived empirically, pinned in `test_charges.py`), and requires an explicit full `conjugates` antiparticle map because charged physical fields like `WВ±` come from rotations, not `conjugate_pair` (so they're absent from `conjugate_map`).

### Quark flavour / CKM (`flavor.py`)

CKM is done the FeynRules-SM way, **never** a symbolic 3Г—3 Yukawa SVD (generic complex Yukawas don't diagonalize in closed form вЂ” nested radicals): work in the quark **mass basis** with diagonal Yukawas and **insert** V into the charged current only (`g/в€љ2 WвЃє Е«_i Оі^ОјP_L V_ij d_j`), leaving neutral currents flavour-diagonal (GIM). `standard_ckm()` builds V from the exact PDG standard parametrization (3 angle externals + 1 phase + 9 complex-internal `V_ij`), unitary as a trig identity so `VвЂ V` simplifies to `1` (a matrix of 9 independent symbols is NOT unitary and would fake Z-FCNCs). CKM elements must be **scalar** parameters (`ParameterSet` is scalar-only; the UFO printer emits an invalid `V[0,1]` subscript for an `IndexedBase`); the 9 `V_ij` export as `type='complex'` internals and round-trip. The 2-gen Cabibbo case IS derivable through the ordinary real `Rotation`/`rotation_2x2` machinery вЂ” `test_ckm.py::TestTwoGenerationGIM` pins the GIM cancellation (mixing in W, exact `cosВІ+sinВІ` diagonality in Z). Worked demo: `examples/sm_ckm.py`.

### Decay widths (`pheno/`) вЂ” Feynman rules в†’ observables

Tree-level **1в†’2 partial widths, total widths and branching ratios** from a built `Model` (`О“ = вџЁ|M|ВІвџ©В·в€љО»(MВІ,mв‚ЃВІ,mв‚‚ВІ)/(16ПЂMВі)`). The `|M|ВІ` path is deliberately **covariant, not explicit 4Г—4 matrices**: Dirac traces go through `sympy.physics.hep.gamma_matrices` (`gamma_trace`/`kahane_simplify` вЂ” ships with SymPy, so still pure-SymPy, no new dependency), contracted with spin/polarization sums and reduced to on-shell `p_iВ·p_j` only at the end. Both routes were prototyped and give identical widths; covariant was chosen for frame-independent `|M|ВІ` and a cleaner path to a later 2в†’2. **The explicit-matrix evaluator lives on as the test oracle** (`tests/test_pheno.py::_oracle_*`), sharing no code with the engine, so no width rests on the covariant path alone.

Three gaps in SymPy's engine, all filled in `pheno/lorentz.py` (each found empirically, not guessed): (1) `gamma_trace` **crashes on mass terms** вЂ” `pМё+m` is a `TensAdd` whose scalar arg has no `.sorted_components()`; `dirac_trace` splits scalar pieces off and applies `Tr[cВ·Iв‚„]=4c`. (2) **No Оів‚…**, so `P_L`/`P_R` are unrepresentable; `reduce_projectors` pushes projectors right (`P_{L,R}Оі^Ој=Оі^ОјP_{R,L}`, `PВІ=P`, `P_LP_R=0`) and uses `Tr[X P]=ВЅTr[X]в€“ВЅTr[XОів‚…]`, **dropping the Оµ term only where it provably vanishes** вЂ” a 1в†’2 final state has just 2 independent momenta and every free index meets a *symmetric* polarization sum, while Оµ is antisymmetric. This is a **hard guard**, not a comment: `reduce_projectors` takes `n_momenta`/`n_free_indices` and raises `NotImplementedError` outside that regime, so a future 1в†’3 fails loudly instead of returning a plausible wrong number (pinned in `test_pheno.py`). (3) `replace_with_arrays` **fails on contracted dummies** (SymPy 1.14, `ValueError: p1(L_0) not found`); `contract_to_dots` walks `TensMul.components`/`.get_indices()`/`.coeff` directly вЂ” the one place touching semi-private tensor API, isolated on purpose and pinned by tests.

Two gotchas that produce *silently wrong numbers*: (1) **a Dirac fermion is two `WeylFermion`s**, so `e_L` and `e_R` are different legs and `extract_fermion_vertices` reports the L and R currents under different keys вЂ” `DecayCalculator(particle_map={eL[i]: e, eR[i]: e, ...})` merges them; omitting it yields two half-channels each missing the other chirality *and* the `g_Lg_R` mass interference. (2) **thresholds**: with symbolic masses `M в‰Ґ mв‚Ѓ+mв‚‚` is undecidable so the channel is kept (correct вЂ” the user may substitute masses that open it), but substituting numbers into a closed channel makes `в€љО»` **imaginary** and poisons the total width and every BR (`hв†’WW` at `m_h=125` is exactly this) вЂ” use `numeric_partial_widths`/`numeric_branching_ratios`, which re-test the threshold after substitution, **not** `numeric(partial_widths(...))`.

`pheno/vertices.py`'s `collect_decay_vertices` unifies the two extraction tracks into one `DecayVertex` list, doing the `P_L`/`P_R` split **once** (every existing caller вЂ” `scripts/export_sm_ufo.py`, `examples/sm_u1x.py`, `examples/sm_vll.py` вЂ” hand-rolls it); a candidate for promotion into core. `pheno/trace.py` is a separate small direct evaluator (в‰¤4 Оі) kept as an *independent* cross-check of the conventions, not on the compute path вЂ” building it out surfaced and fixed a real bug there: `Оі^ОјОі^Ој` collapses to a `Pow`, which its `Mul`-only dispatch missed, returning `4В·gamma(0)**2` instead of `4g^{ОјОј}`. Implemented vertex types: **SSS, FFS, FFV, VVS**; `VSS`/`VVV` carry `p(П†)` momentum tags from `to_momentum_space` and raise (derivative couplings are not yet squared). Worked model: `examples/sm_decays.py` (reproduces `О“(Zв†’ОЅОЅМ„)=0.168`, `О“(Zв†’П„П„)=0.084`, `О“(Wв†’П„ОЅ)=0.227` GeV, `О“(hв†’П„П„)=0.26` MeV vs PDG).

### MadGraph round-trip (the exported UFO actually computes right)

`scripts/export_sm_ufo.py` + `scripts/madgraph_roundtrip.py` export an EW+leptons SM UFO and compare cross sections to MadGraph's stock `sm` at a matched parameter point (`e+e-в†’Ој+Ој- = 2.788 pb`, `e+e-в†’W+W- = 19.50 pb`, both to MC precision; `docs/benchmark.md`). Not in CI (each launch compiles Fortran). MG5 lives at `~/.local/mg5dl/MG5_aMC_v3_7_2` and runs on Python 3.13. **Three UFO-export issues this surfaced** (first two fixed in the library + regression-tested, invisible to unit tests but fatal in MG): (1) generated UFO files must use **absolute** imports (`from object_library import вЂ¦`, not `from .object_library`) вЂ” MG runs them as standalone scripts for param-card generation; `verify/ufo_roundtrip.py` + the test `_import_ufo` helpers now load a UFO the MG way (dir on `sys.path`, import submodules by bare name). ### SU(3) / QCD vertex dynamics

The group theory (`SU3` Gell-Mann generators/structure constants, `groups/gauge.py`) and the vertex-extraction machinery (`Dmu`, `fermion_gauge_current`, `cubic_couplings`/`quartic_couplings`) were already fully generic over gauge group вЂ” no SU(3)-specific code was needed there. What closed the gap: `examples/sm_scalar_gauge.py` now includes a quark sector (SU(2) doublet `QL` + `SU3c`-triplet singlets `uR`/`dR`, flavor-generic `Yu`/`Yd` Yukawas mirroring the lepton pattern вЂ” 3 generations, undiagonalized, no CKM: CKM mixing is physically orthogonal to SU(3) vertex dynamics and a fully generic 3-generation complex-Yukawa SVD doesn't resolve in closed symbolic form) and a gluon field; `export/ufo/vvvv.py`'s `assemble_vvvv` builds the 3-structure VVVV1/2/3 decomposition of the 4-boson self-coupling that `yangmills.py`'s `quartic_couplings()` docstring had deferred as unbuilt "Phase 5" (derived by direct functional differentiation of `-ВјF^aF^a`, cross-checked against both SU(2) and SU(3) in `tests/test_yangmills.py`); `export/ufo/writer.py` emits real color-tensor strings (`T(3,1,2)` for qqg, `f(1,2,3)` for ggg, three `f*f` structures for gggg) instead of the hardcoded singlet `'1'` вЂ” gluons are exported as **one** UFO particle repeated (color summed via the tensor string), never as 8 separate weak-basis components.

The notebooks are tracked through a `nbstripout` git filter (`.gitattributes`, `*.ipynb filter=nbstripout diff=ipynb`), configured with `--keep-output` so real outputs (plots, printed Feynman rules) stay in git history and in diffs вЂ” only `execution_count` and volatile per-cell metadata get normalized away on `git add`/`git diff`, so re-executing a notebook without changing its actual results produces no diff noise. A fresh clone needs to activate the filter once (`.gitattributes` alone only declares the pattern, it doesn't install the filter driver):

```bash
nbstripout --install --attributes .gitattributes
git config filter.nbstripout.clean "$(git config filter.nbstripout.clean) --keep-output"
git config diff.ipynb.textconv "$(git config diff.ipynb.textconv) --keep-output"
```

(`nbstripout --install` itself does not persist CLI flags like `--keep-output` into the stored filter command вЂ” they have to be appended to `filter.nbstripout.clean`/`diff.ipynb.textconv` by hand, as above.)
